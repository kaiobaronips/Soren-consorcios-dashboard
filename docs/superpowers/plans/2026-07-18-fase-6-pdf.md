# Fase 6 — Base de Produtos (PDF): Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Área administrativa "Base de Produtos": upload de PDFs de tabelas de consórcio, extração de texto/tabelas com confiança por campo, OCR de fallback, revisão humana lado a lado obrigatória e publicação com dedup/atualização — nunca publicando nada sem validação humana.

**Architecture:** Pipeline `upload → storage seguro → hash → extração de texto → identificação de produtos → validação → revisão humana → publicação`. Parsing de produtos é **puro e testável** (`src/lib/pdf/parse-products.ts`: texto → candidatos com confiança por campo). Extração de PDF (pdfjs-dist) e OCR (tesseract.js) rodam **no servidor** (Server Action / route handler com runtime nodejs). Estado do documento na tabela `product_documents` (status: uploaded/processing/review_required/completed/failed) com `extraction_log` JSONB. Publicação reusa a chave de dedup da Fase 2.

**Nota de escopo:** a Fase 5 (CRM/Kanban/dashboard) foi CANCELADA por decisão do usuário em 2026-07-18 — o sistema é de atendimento único. A ordem passou a ser F4 → F6 → F7. Registrar em PLANS.md na task de gate.

### Task 1: Parsing puro de produtos a partir de texto (o coração testável)

**Files:**
- Create: `src/lib/pdf/parse-values.ts`, `src/lib/pdf/parse-values.test.ts`, `src/lib/pdf/parse-products.ts`, `src/lib/pdf/parse-products.test.ts`

**Interfaces:**
- Produces:

```ts
// parse-values.ts — normalização pt-BR pura
export function parseBrlMoney(raw: string): string | null;   // "R$ 600.000,00"/"600.000,00" → "600000.00"; inválido → null
export function parseBrlPercent(raw: string): string | null; // "26,8%"/"26,8" → "26.800"; inválido → null
export function parseIntSafe(raw: string): number | null;    // "240"/"240x" → 240; inválido → null

// parse-products.ts — heurística de tabela pura
export type ExtractedField<T> = { value: T | null; confidence: number; raw: string | null }; // confidence 0–100; value null = PENDENTE
export type ExtractedProduct = {
  page: number;
  productName: ExtractedField<string>;
  productCode: ExtractedField<string>;
  creditAmount: ExtractedField<string>;
  termMonths: ExtractedField<number>;
  totalAdministrationFeePercent: ExtractedField<string>;
  regularInstallmentAmount: ExtractedField<string>;
  first12InstallmentAmount: ExtractedField<string>;
  overallConfidence: number;
  issues: string[]; // mensagens legíveis: "coluna Taxa não identificada na página 2"
};
export type PageText = { page: number; lines: string[] };
export type ColumnMapping = Partial<Record<"productName"|"productCode"|"creditAmount"|"termMonths"|"totalAdministrationFeePercent"|"regularInstallmentAmount"|"first12InstallmentAmount", number>>; // índice da coluna, para mapeamento manual
export function extractProductsFromText(pages: PageText[], manualMapping?: ColumnMapping): { products: ExtractedProduct[]; log: string[] };
```

Heurística (documentar no código): detectar linha de cabeçalho por palavras-chave (crédito/carta, prazo, taxa, parcela, código, produto — case/acento-insensitive); inferir colunas por posição/ordem; linhas de dados = linhas com ≥2 valores monetários ou padrão código+números; `manualMapping` sobrepõe a inferência. Campo não reconhecido → `value: null`, confidence 0, issue registrada. Validações (§8.9): carta > 0; prazo 1–600; taxa 0–100; parcela > 0 e < carta — violação zera a confiança do campo e vira issue.

- [ ] **Step 2:** FAIL → implementar → PASS.
- [ ] **Step 3:** `pnpm lint && pnpm typecheck && pnpm test` verdes → Commit: `feat: parsing puro de produtos de PDF com confiança por campo e mapeamento manual`
